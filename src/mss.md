## 场景一：复杂日志查询解释（Agent核心）

### 业务痛点还原
> 客户在月报里看到“攻击日志top10”，但在日志检索中心自己查，数量对不上。因为后台查询时过滤了资产组IP、资源组IP等复杂条件，客户不知道这些隐藏的过滤逻辑。

### 技术实现方案

**核心流程**：
```
用户问：“为什么我查的攻击日志数量比月报少？”

Agent理解意图 → 调用get_report_query_conditions工具（获取月报的查询条件）
                → 调用get_user_query_conditions工具（获取用户自己的查询条件）
                → 对比两个条件，找出差异
                → 生成SQL，让用户去日志中心查询
```

**需要新增的工具**：
1. `get_report_query_conditions(report_id, metric_name)`：返回月报生成时用的查询条件（资产组过滤、时间范围等）
2. `get_user_query_conditions(user_id, query_time_range)`：获取用户在日志中心的查询条件
3. `generate_sql_with_conditions(conditions)`：根据条件生成可在日志中心执行的SQL

**数据模拟方案**：
你不需要真的接入ClickHouse，可以用JSON模拟：
```python
# 模拟月报查询条件
report_conditions = {
    "time_range": "2026-02-01 00:00:00 to 2026-02-28 23:59:59",
    "asset_groups": ["group_A", "group_B"],  # 只统计这两个资产组
    "resource_groups": ["prod", "staging"],  # 只统计这两个资源组
    "attack_types": ["sql_injection", "xss"],  # 只统计这两种攻击
    "excluded_ips": ["192.168.1.0/24"]  # 排除内网IP
}

# 模拟用户查询条件
user_conditions = {
    "time_range": "2026-02-01 to 2026-02-28",  # 可能格式不一样
    "asset_groups": [],  # 用户没选，默认所有
    # ... 其他条件
}
```

**Agent的对话示例**：
```
用户：为什么我查的SQL注入攻击数量比月报少？

Agent：我查了一下，月报里的SQL注入攻击统计有两个过滤条件：
1. 只统计了资产组A和B的资产（您的查询可能统计了所有资产）
2. 排除了内网IP段192.168.1.0/24的告警

这是月报使用的完整查询条件，您可以复制到日志中心重新查询：
SELECT COUNT(*) FROM attack_logs 
WHERE attack_type='sql_injection' 
AND asset_group IN ('group_A', 'group_B')
AND src_ip NOT LIKE '192.168.1.%'
AND timestamp BETWEEN '2026-02-01' AND '2026-02-28'
```

### 这个场景的面试价值
- **展示业务理解**：能讲出SaaS平台客户对账的真实痛点
- **展示Agent设计**：多个工具协同完成复杂任务
- **展示工程思维**：条件对比、SQL生成、格式归一化

## 场景二：安全评分解读（RAG增强）

### 业务痛点还原
> 月报里给客户打了一个安全评分（比如78分），但没说这个分怎么算的。客户问“为什么我只有78分”，客服要翻一堆文档才能回答。

### 技术实现方案

**核心流程**：
```
用户问：“这个月的安全评分78分是怎么算的？”

RAG流程：
1. 检索评分定义文档（PDF/Markdown）
2. 找到评分计算公式和相关说明
3. LLM根据文档内容和用户问题生成解释
```

**需要准备的文档**（模拟）：
创建一个`scoring_definition.md`，内容示例：
```markdown
# 安全评分计算方法（v2.1）

## 评分维度及权重
- 攻击次数（30%）：统计周期内检测到的总攻击次数
  - 0-1000次：满分30
  - 1000-5000次：20分
  - 5000次以上：10分
- 攻击成功次数（30%）：实际成功的攻击次数
  - 0次：30分
  - 1-5次：20分
  - 5次以上：10分
- 告警数量（20%）：生成的告警总数
- 漏洞风险（10%）：根据高危漏洞数量计算
- 弱密码（5%）：存在弱密码的资产比例
- 暴露面风险（5%）：开放的高危端口数量

## 计算公式
总分 = 攻击次数得分 + 攻击成功得分 + 告警得分 + ...（详细说明）
```

**RAG实现要点**：
1. **文档切片**：按章节切分，保留标题层级
2. **检索增强**：用户问题+评分78分这个具体数值，一起作为检索上下文
3. **回答生成**：让LLM不仅解释公式，还要结合78分反推可能的原因

**对话示例**：
```
用户：这个月评分78分，是怎么算的？

Agent：根据我们的评分文档（v2.1版），安全评分由6个维度加权计算：

您这个月78分，主要扣分项是：
1. 攻击成功次数：有3次成功攻击（扣了10分）
2. 高危漏洞：存在2个高危漏洞（扣了5分）

具体明细：
- 攻击次数（满分30）：您得了28分（攻击次数在1000-5000区间）
- 攻击成功（满分30）：您得了20分（3次成功攻击）
- 告警数量（满分20）：您得了18分
- 漏洞风险（满分10）：您得了5分（2个高危漏洞）
- 弱密码（满分5）：满分5分
- 暴露面（满分5）：满分5分
合计：28+20+18+5+5+5=101分？等等，我重新算一下...
```

（这个例子说明：需要让LLM计算准确，或者提前做好评分明细数据的模拟）

### 这个场景的面试价值
- **展示RAG落地能力**：不是简单跑Demo，而是有实际业务场景
- **展示文档处理**：切片策略、检索优化
- **展示复杂逻辑**：结合文档+具体数值做解释
